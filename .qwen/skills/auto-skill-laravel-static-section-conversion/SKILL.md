# Convert a DB-driven section to a static React component

When a Laravel + Inertia/React feature has an admin CRUD that is rarely
edited, is half-broken, or is no longer worth the maintenance cost, the
cleanest simplification is to:

1. **Capture the current copy** from the database (the live text + image
   paths) so nothing visible to the public changes.
2. **Delete the admin layer** (controller, routes, React pages, model
   references in route closures, sidebar links).
3. **Strip the props** from the public component, hardcoding the same
   title / description / image.
4. **Verify** with `php -l`, `npm run build`, and `php artisan route:list`
   to confirm no dead references remain.

This skill is the procedure for steps 1–4 in one shot. Pair with
`laravel-feature-references-investigation` if you first need to confirm
the feature is safe to convert (broken admin vs. half-wired vs.
deliberately live).

## Before you start: check the working tree first

**The user (or a prior turn) may have already partially executed this
skill.** The first command you run should be `git status --short` to see
what's already in the working tree.

Common pre-existing states you might find:

- `D app/Http/Controllers/Admin/FooController.php` (controller already
  deleted, but routes and component still reference it)
- `M routes/web.php` (admin routes already removed, closure prop still
  there — or vice versa)
- `M resources/js/Components/Home/Foo.jsx` (component already
  hardcoded, parent page still passes a prop it doesn't use)
- `D resources/js/Pages/Admin/Foo/*.jsx` (admin pages already deleted)

**Treat the working tree as the starting point, not the assumed clean
state.** Your plan should be a **diff against the working tree**, not
against HEAD. Files already in the desired state get a "✅ already in
tree — none" row in your report, and you should not re-edit them.

This is also the moment to detect **other in-progress work** the user
is touching alongside the conversion. The git status may show
modifications to `MepFooter.jsx`, `Keyhighlights.jsx`, `About.jsx`,
unrelated controllers, etc. These are NOT part of the conversion —
**do not edit them in this turn** even if they look related. Note them
in the report as "outside scope, untouched."

## When to use

The user asks:
- "Remove the admin CRUD for X"
- "Make the [homepage section] static / hardcoded"
- "Stop letting admin edit X — just hardcode the current text"
- "I never edit X anyway, simplify it"
- Or after a `laravel-feature-references-investigation` verdict, the user
  asks to demote the feature from a DB table to a static component

## Procedure

### 1. Capture the current copy BEFORE deleting anything

This is the most important step. The user will lose the ability to edit
this content once the CRUD is gone, so the exact text and image that
was rendering must be captured first.

```bash
php artisan tinker --execute="echo \App\Models\Foo::all()->toJson(JSON_PRETTY_PRINT);"
```

For each row, capture:
- `title` (will be hardcoded into the JSX)
- `description` / `body` / `content` (longest copy field)
- `image` (file path relative to `storage/`, or full URL if external)
- `button_text` and `button_link` (note if NULL — often they are)
- `is_active` (used to filter the public render — irrelevant once static)

**Always quote the JSON back to the user** in the report so they can
confirm the values you captured match what's currently visible on the
public site.

### 2. Find all references to the feature in code

Run in parallel:

```text
grep_search "Foo|FooController|admin/foo" in:
  - app/Http/Controllers/        (controllers — including non-admin ones)
  - app/Models/                   (model)
  - resources/js/                 (frontend, but EXCLUDE build/ and public/build/)
  - routes/                       (routes)
  - database/seeders/             (seeders)
```

**Exclude `build/`, `public/build/`, and `node_modules/` from the
grep** — these are compiled artifacts. The `build/assets/Home-*.js`
bundle will contain every `CtaBanner` reference that ever lived in
source. Those are NOT live references; they get regenerated by
`npm run build`. Counting them will lead you to "fix" non-issues.

**Watch for inlined CRUD methods.** In this project, several
"small" features had their admin actions inlined into another
controller instead of getting their own `FooController.php`. For
example, cta-banner's `index/edit/update/destroy` lived on
`HeroController` as `ctaIndex/ctaEdit/ctaUpdate/ctaDestroy`. The grep
will find these as:

```php
Route::get('cta-banner', [HeroController::class, 'ctaIndex'])->name('cta-banner.index');
```

So the route file's `use App\Http\Controllers\Admin\CtaBannerController;`
may be gone while the routes themselves still exist, OR the routes may
be inlined with NO `CtaBannerController` import. **Always grep for
the route URI/prefix** (`admin/cta-banner`, `admin/foo`) in
`routes/web.php`, not just the class name. The class name is the
wrong anchor when CRUD is inlined.

Build a complete file list before editing. For each file, classify it
as: **delete**, **modify**, or **leave alone**. Don't skip this — the
"orphan import" in `routes/web.php` (e.g. `use App\Models\Foo;` after
the only query is removed) is the most common miss.

### 3. Check the public component's existing implementation

Read the public component file (e.g. `resources/js/Components/Home/Foo.jsx`)
in full. Note:

- Which props it accepts from the parent page
- Whether it has **fallback strings** (often `prop?.field || 'default text'`)
  — these are clues that the field is rarely set and the hardcoded text
  is what the user actually sees
- Whether it uses a prop image or a hardcoded one — the DB's `image`
  field may have been captured but never wired up
- Whether the button is rendered or skipped (if `button_text` is NULL
  in the DB, the component probably never renders the button)

This affects the static version's diff size. A component that already
has hardcoded fallbacks for everything is a 5-line edit. One that
heavily uses dynamic props is a 20-line edit.

### 4. Plan the deletion + modification set

Build a table:

| File | Action | Reason |
|---|---|---|
| `app/Http/Controllers/Admin/FooController.php` | DELETE | admin CRUD, no longer needed (skip if absent) |
| `app/Http/Controllers/Admin/OtherController.php` | MODIFY (remove inlined methods) | if the CRUD is inlined as `fooIndex/fooEdit/...` methods on another controller |
| `resources/js/Pages/Admin/Foo/{Index,Edit,Create}.jsx` | DELETE | admin CRUD pages (skip if absent) |
| `resources/js/Pages/Admin/Foo/` directory | DELETE | (after files removed) |
| `routes/web.php` | MODIFY | remove `use App\Models\Foo;`, remove 4–7 admin routes (whether `Route::resource('foo', FooController::class)` or 4 inlined `Route::get/put/delete` calls), remove prop in Home closure |
| `resources/js/Components/Home/Foo.jsx` | MODIFY | strip props, hardcode copy |
| `resources/js/Pages/Home.jsx` (or parent page) | MODIFY | drop prop pass-through |
| `app/Models/Foo.php` | LEAVE | user said don't drop the model |
| `database/migrations/*_create_foos_table.php` | LEAVE | user said don't drop the migration |
| `database/seeders/*FooSeeder*` | LEAVE | seeding still works if run manually |

**Always ask the user** whether the model and migration should be left
alone or fully removed. They may want to keep the table for historical
data, or they may want a clean break. The default is "leave alone" for
safety, but confirm.

**If the working tree already has the file in the desired state** (e.g.
`D app/Http/Controllers/Admin/FooController.php` from a prior turn),
add a "✅ already in tree — none" row to the table. This tells the user
the change is done without re-issuing the same edit.

### 5. Don't forget the admin dashboard / sidebar

The feature may appear in:

- `resources/js/Pages/Dashboard.jsx`'s `getModules()` array (admin home)
- `resources/js/Layouts/AuthenticatedLayout.jsx` (top nav)
- `resources/js/Layouts/MepLayout.jsx` (any admin nav in there)
- A `KeyHighlight` or `Sidebar` component imported by admin pages

`grep_search` for the feature name in `resources/js/` to catch all of
these. If the feature is **already not linked from anywhere** (only
reachable by typing the URL), the sidebar/dashboard step is a no-op —
just confirm that in the report so the user trusts nothing was missed.

### 6. The "check if any other route references it" check

Even after the feature is removed from `routes/web.php`, search for
**named-route references** (e.g. `route('admin.foo.index')`) elsewhere:

```text
grep_search "admin.foo|admin/foo" in resources/js/, app/
```

If anything still calls the named route, it will 500. Catch this
BEFORE deleting the route definition.

### 7. The static-component edit pattern

The new JSX should look like this (example, real shape depends on
original component):

```jsx
import FadeIn from "@/Components/FadeIn";
import { Link } from "@inertiajs/react";

// Captured from the database on YYYY-MM-DD before deletion.
// Update by editing this file directly, not from admin.
const TITLE = "Trusted by Leading Brands in Cambodia";
const DESCRIPTION = "Master MEP has worked with leading brands, ...";
const BACKGROUND_IMAGE = "/HeroSection/heroSection.png";

export default function Foo() {  // no props
  return (
    <section
      className="relative py-20 bg-cover bg-center text-white text-center"
      style={{ backgroundImage: `url(${BACKGROUND_IMAGE})` }}
    >
      <div className="absolute inset-0 bg-black/60" />
      <FadeIn className="relative z-10 max-w-2xl mx-auto px-6">
        <h2 className="text-3xl md:text-4xl font-bold mb-4">{TITLE}</h2>
        <p className="text-sm md:text-base opacity-80 mb-6">{DESCRIPTION}</p>
        <Link
          href="/contact"   // hardcoded destination, not from DB
          className="inline-block px-6 py-2.5 bg-[#1A3A5C] hover:bg-[#2E5C8A] text-white text-sm font-medium rounded-lg transition"
        >
          Contact Us
        </Link>
      </FadeIn>
    </section>
  );
}
```

Key decisions:
- **Drop all props.** The component should accept none.
- **Hardcode the destination for the button** (e.g. `/contact`). Don't
  leave a placeholder for an `href` prop — the whole point is that the
  destination is now in the code, not the DB.
- **Add a comment** with the capture date. Six months from now when
  someone wants to change the copy, the comment tells them where to look.
- **Decide whether to add a button at all.** The DB row may have
  `button_text = NULL` (meaning the public version never showed a
  button). Adding a "Contact Us" link to `/contact` is a small UX win
  but is a behavior change. **Ask the user** before adding a button
  the public site didn't have before.

### 8. The route file edit pattern

`routes/web.php` typically needs three small edits:

```diff
- use App\Models\Foo;
  use App\Models\Service;
```

```diff
  Route::get('/', function () {
      return Inertia::render('Home', [
          'heroes'    => Hero::...,
-         'foo'       => Foo::where('is_active', true)->latest()->first(),
          'services'  => Service::...,
      ]);
  })->name('home');
```

```diff
  Route::middleware(['auth', 'admin'])->prefix('admin')->name('admin.')->group(function () {
      Route::resource('bar', BarController::class);

-     Route::get('foo',                   [FooController::class, 'index'])->name('foo.index');
-     Route::get('foo/{foo}/edit',         [FooController::class, 'edit'])->name('foo.edit');
-     Route::put('foo/{foo}',              [FooController::class, 'update'])->name('foo.update');
-     Route::delete('foo/{foo}',           [FooController::class, 'destroy'])->name('foo.destroy');
-
      Route::resource('baz', BazController::class);
  });
```

Three removals: import, public prop, admin routes. The order matters
for syntax-correctness — always remove the import first, then the prop
(relying on it), then the routes (which depended on the controller that
you've already deleted).

**Inlined-CRUD variant.** If the project inlines the CRUD on another
controller (e.g. cta-banner methods on `HeroController`), the route
block looks like:

```php
Route::get('foo',                   [HeroController::class, 'fooIndex'])->name('foo.index');
Route::get('foo/{foo}/edit',         [HeroController::class, 'fooEdit'])->name('foo.edit');
Route::put('foo/{foo}',              [HeroController::class, 'fooUpdate'])->name('foo.update');
Route::delete('foo/{foo}',           [HeroController::class, 'fooDestroy'])->name('foo.destroy');
```

The class is `HeroController`, which you are NOT deleting — it's used
by the real `Route::resource('hero', HeroController::class)` line right
above. So you cannot just remove the import. You must surgically
remove the 4 inlined route lines only, leaving the `HeroController`
import and resource route intact. The corresponding `fooIndex/fooEdit/
fooUpdate/fooDestroy` methods in `HeroController` should also be
removed in the same turn (they're orphan methods after the routes
go, and the `laravel-code-review` skill will flag them as dead code).

**If the working tree already shows the route block removed** in
`git diff routes/web.php` (i.e. the user's previous turn already did
this), do not re-apply. The "before you start" check above is what
catches this. The single highest-leverage command in this whole skill
is `git status --short` at the very start.

### 9. Verification — three checks, in order

```bash
# 1. PHP syntax of the only PHP file changed
php -l routes/web.php
# Expected: "No syntax errors detected"

# 2. Frontend build — catches dead React imports / broken JSX
npm run build
# Expected: "✓ built in X.XXs", no errors

# 3. No leftover route definitions
php artisan route:list --columns=method,uri,name | findstr /I "foo"
# Expected: empty (or only the unrelated hits, e.g. "food" in some
# other context — read the output, don't assume empty means success)
```

The third check is the highest-signal one. If `findstr foo` returns
nothing, you have a clean removal.

### 10. Report format

1. **Current data to preserve** — the JSON dump of the rows (with
   `button_text`/`button_link`/`is_active` clearly marked as `null` if
   applicable).
2. **Current component** — show the JSX *before* the edit, so the user
   can confirm what hardcoding will replace.
3. **Working-tree pre-state** — what was already in the working tree
   from a prior turn (e.g. controller already deleted, routes already
   removed). This sets expectations that your edits this turn may be
   fewer than the full procedure.
4. **Files changed table** — file | action | one-line reason. Include
   "✅ already in tree — none" rows for files in the desired state.
5. **Per-file diff** — only the +/- lines, with 3 lines of context.
6. **Verification results** — output of the three checks.
7. **What's NOT changed** — model, migration, seeded data, sidebar (if
   already not linked), AND any unrelated in-progress modifications
   visible in `git status` that are outside this task's scope. This
   builds trust that you didn't accidentally delete or rewrite
   something the user wanted to keep.

## What NOT to do

- Don't capture the copy from memory — always run `php artisan tinker`
  on the actual DB. Copy may have been edited since the migration.
- Don't drop the model or migration without explicit user confirmation
  even if the user said "remove the feature" — DB drops are
  destructive, file deletes are recoverable from git.
- Don't auto-add a button that wasn't in the original component. If
  `button_text` was NULL and the original JSX had no button, the static
  version should also have no button **unless the user explicitly says
  "with button" or similar.** When the user does opt in, a
  `Contact Us → /contact` hardcoded link is the standard pick.
- Don't add a `// TODO: re-add admin CRUD` comment. The point of
  conversion is to delete the CRUD. If the user changes their mind,
  they can `git revert` the deletion commit; a TODO comment is just
  noise.
- Don't skip the `route:list` verification. PHP's autoloader won't
  catch a missing controller class until runtime, and the user may not
  hit the broken route in casual testing.
- Don't reorder route removals across the three sections. Import →
  prop → routes is the order that keeps `routes/web.php` syntactically
  valid at every intermediate step (in case the user wants to commit
  halfway).
- Don't apply changes without first showing the user the captured
  data + planned file list. The user has a strong preference (across
  every prior task in this project) for "show me the plan before
  deleting" — even when the deletions seem safe.
- **Don't count `build/assets/*.js` grep hits as live references.**
  They are compiled artifacts. `npm run build` will rewrite them.
- **Don't grep only for the controller class name** when looking for
  the CRUD in `routes/web.php` — the CRUD may be inlined onto another
  controller, in which case the class name won't appear at all. Grep
  for the route URI prefix instead.
- **Don't touch unrelated in-progress modifications** visible in
  `git status`. The user may be working on several things in parallel;
  the conversion is one of them. If `MepFooter.jsx`,
  `Keyhighlights.jsx`, or other files show modifications unrelated to
  the section being converted, note them in the "What's NOT changed"
  section of the report and leave them alone.

## Why this procedure

- The user is **losing the ability to edit content** when the CRUD is
  deleted. Capturing the current copy is the only safety net. Git
  history is a backup, but `git log` on a config file is a worse
  authoring experience than editing a JSX constant.
- The DB row often contains more fields than the public component
  uses (e.g. `cta_banners.image` was never wired up to the JSX
  background). Reading the original component in full before
  deciding the static edit size prevents capturing irrelevant fields.
- The "import → prop → routes" deletion order keeps the file
  syntactically valid at every intermediate state, which is what
  you want if a user decides mid-edit to revert a specific change.
- Verification with all three tools catches three different bug
  classes: `php -l` catches syntax errors, `npm run build` catches
  broken React imports, `route:list | findstr` catches orphan route
  references that would 500 at runtime.
- Hardcoding with a `const` and a `// captured on YYYY-MM-DD` comment
  makes the file self-documenting: future maintainers see when the
  copy was captured, and that any later change should be made by
  editing the file rather than the (no-longer-existing) admin form.

## How to apply

Use this skill when the user asks to remove a DB-driven section's
admin CRUD and replace it with a hardcoded version. The default
workflow is:

1. `git status --short` to see what's already in the working tree.
2. `php artisan tinker` to dump the current rows (still capture — the
   DB is the source of truth even if the JSX is already hardcoded,
   because you want the captured values to match what was on the
   screen, not what someone might have written in a stale comment).
3. Read the public component in full + grep for all references
   (excluding `build/` and `node_modules/`).
4. Build a delete/modify/leave-alone table. Mark files that are
   already in the desired state with "✅ already in tree — none".
5. Show the plan + captured data + per-file diffs.
6. Wait for the user to confirm.
7. Delete files, edit route file, edit component, edit parent page.
8. Verify with `php -l`, `npm run build`, `route:list | grep`.

Pair with `laravel-feature-references-investigation` if the feature
is half-broken and you need to confirm the conversion is the right
call (vs. fixing the CRUD). Pair with `laravel-code-review` if the
conversion is part of a broader cleanup pass and the user wants
findings on other features too.
